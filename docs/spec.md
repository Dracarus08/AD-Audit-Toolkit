# AD-Audit-Toolkit. Design Spec

## Problem

Active Directory drifts. Accounts go stale, service accounts collect password-never-expires flags, and privileged groups grow one "temporary" member at a time. Most small and mid-size shops have no scheduled way to see this drift. The first thing an attacker does after initial access is enumerate exactly these weaknesses. A defender should see them first.

## Who cares

- Sysadmins who own AD and want a repeatable hygiene check.
- Security engineers who need identity findings without deploying a commercial tool.
- Hiring managers reading this repo. It shows I understand AD as an attack surface, not just a directory.

## What it does

A PowerShell module with one entry point, `Invoke-ADAudit`. It collects users, computers, groups, and password policy, runs ten hygiene checks, and produces a scored HTML report plus structured finding objects.

Checks:

| ID | Check | Default severity |
|----|-------|-----------------|
| STALE-USERS | Enabled users with no logon in 90 days | Medium |
| PWD-NEVER-EXPIRES | Enabled users whose password never expires | Medium, High if privileged |
| PWD-NOT-REQUIRED | Accounts flagged PASSWD_NOTREQD | High |
| OLD-PASSWORDS | Enabled users with passwords older than 365 days | Low |
| PRIV-SPRAWL | Privileged groups above membership thresholds | High |
| PRIV-INACTIVE | Privileged members inactive for 60 days or disabled | High |
| KERBEROASTABLE | User accounts carrying SPNs, worse when privileged | High |
| UNCONSTRAINED-DELEG | Non-DC objects trusted for unconstrained delegation | High |
| DORMANT-COMPUTERS | Enabled computer objects silent for 120 days | Low |
| WEAK-PWD-POLICY | Domain policy below baseline (length, lockout, reversible encryption) | Varies |

## Modes

- **Live mode** reads a real domain through the ActiveDirectory module. Read-only. No writes, ever.
- **Demo mode** (`-Demo`) loads a bundled synthetic domain of about 250 objects with seeded findings. It runs on any machine in under 60 seconds with no domain and no RSAT. Reviewers use this. The dataset is generated by `demo/New-DemoDataset.ps1` and every object in it is fake.

Both modes normalize into the same internal shape, so every check runs identically against either source.

## Tech stack

Windows PowerShell 5.1 compatible module. No external dependencies at runtime. Pester 5 for tests. GitHub Actions for CI with PSScriptAnalyzer linting, tests, and a demo-mode smoke run.

## Framework mapping

Findings map to MITRE ATT&CK techniques attackers use against these weaknesses: T1078 Valid Accounts, T1558.003 Kerberoasting, T1550 Use Alternate Authentication Material, T1110 Brute Force, T1098 Account Manipulation. The point of the mapping is to justify each check in attacker terms.

## Done means

1. `Invoke-ADAudit -Demo` produces the HTML report on a clean machine in under 60 seconds.
2. All checks covered by Pester tests against the seeded dataset.
3. Real screenshots of the report captured from a real run.
4. README to standard. Defense pack written. gitleaks clean.
